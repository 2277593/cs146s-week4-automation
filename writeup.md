## 工作流程摘要
### 核心設計概念
請用 3 到 5 句話說明你的工作流程。重點放在：流程怎麼開始、任務怎麼被完成，以及結果怎麼被呈現。

工作流程從使用者在 Claude Code 輸入 `/do-next-task` 開始，不需要任何其他輸入。Agent 自動讀取 `docs/TASKS.md`，挑選最早的可執行未完成子任務，識別相關檔案後進行實作，並在完成後立即執行測試驗證。每次執行結束後，agent 將本次工作內容、測試結果與下一步建議寫入 `reports/latest.html`，並透過 macOS 系統通知告知使用者報告已就緒。整個流程一個指令觸發，全程無需人工介入。

### 具體實作內容
請描述你使用了哪些技術、分別做了什麼。

**CLAUDE.md**：撰寫專案操作手冊，涵蓋常用指令（`uv run make test`、`uv run make lint` 等）、後端架構說明（FastAPI + SQLAlchemy + SQLite、dependency injection 測試模式）、以及 agent 應遵循的完整工作流程規則，讓每個新的 Claude Code 實例無需重新探索專案即可立即上工。

**`.claude/commands/do-next-task.md`**：定義主要自動化指令的完整執行邏輯，包含七個明確步驟：讀取任務清單 → 選任務（依最小編號、跳過被擋任務、只取第一個子任務）→ 識別相關檔案 → 實作 → 執行測試 → 產生 HTML 報告 → 發送 macOS 通知。報告格式規格也寫在指令檔中，確保每次輸出結構一致。

**`.claude/commands/add-task.md`**：定義輔助指令，讓使用者用自然語言一句話新增任務。指令會自動判斷現有任務編號、推斷 2–5 個具體可執行的子任務，並以與原始格式完全一致的方式附加至 `docs/TASKS.md` 末尾。

## 反思

### 前後比較
你覺得加上你的自動化工作流之後的差異在哪裡？

加上自動化工作流之前，每次要推進任務都需要自己判斷「現在該做什麼」、手動執行測試、再手動整理結果，容易漏步驟或忘記跑 lint。加上之後，整個過程變成一個指令：任務選擇、實作、驗證、報告一氣呵成，且每次的輸出格式都一樣，方便比對不同執行回合的差異。最直接的感受是「不需要再花心力管流程本身」，可以把注意力放在確認結果是否正確。

### 最有價值的部分
你設計的工作流程裡，最後證明最有用的是哪些？

最有價值的是 **CLAUDE.md 中明確的任務挑選規則**。如果只寫「挑下一個任務」，agent 在面對有多個未完成任務、或某個任務一半完成一半沒完成的情況時，行為會不穩定。明確寫出「最小編號優先 → 跳過被擋的 → 只取第一個子任務 → 太大則縮範圍」之後，每次執行的結果都是可預期的，也更容易除錯。其次是**報告的五段固定結構**，讓每次執行後都能快速確認「做了什麼、測試過了嗎、有沒有問題、下一步是什麼」，不需要翻 terminal log。

### 下一步想改進什麼
如果再多一點時間、懂更多技術，你最想補強或延伸哪個部分？

最想補的是**報告的歷史累積機制**。目前 `reports/latest.html` 每次覆寫，無法回頭查看前幾次執行的狀態。理想做法是以時間戳命名（如 `reports/2026-04-09T20:00.html`），並在 `latest.html` 加入歷史連結列表。其次是**跨平台通知支援**：目前通知只在 macOS 有效，在 Linux 可以改用 `notify-send`，讓工作流在不同環境下都能完整運作。
